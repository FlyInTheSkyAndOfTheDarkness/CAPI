# CAPI — конверсии из CRM в рекламные платформы

Мульти-тенантная SaaS-платформа: принимает события из **amoCRM** и **Битрикс24**
(смена этапа воронки у лида/сделки) и отправляет серверные конверсии в
**Meta Conversions API** и **TikTok Events API** с SHA-256 хешированием
персональных данных (email, телефон).

## Архитектура

```
apps/
├── api/   NestJS + Prisma (PostgreSQL) + BullMQ (Redis)
│   ├── auth/          JWT-аутентификация, регистрация с авто-созданием воркспейса
│   ├── workspaces/    мульти-тенантность (заголовок X-Workspace-Id)
│   ├── connections/   подключения CRM: amoCRM (OAuth 2.0 или долгосрочный токен),
│   │                  Битрикс24 (входящий вебхук)
│   ├── destinations/  рекламные платформы: Meta Pixel, TikTok
│   ├── mappings/      правила: этап воронки CRM → событие конверсии
│   ├── webhooks/      публичный приём вебхуков CRM (секрет в URL)
│   ├── delivery/      очереди BullMQ: обработка событий + отправка с ретраями
│   └── logs/          логи доставки, статистика для дашборда
└── web/   React 19 + Vite + Tailwind CSS 4 — личный кабинет (RU)
```

Дашборд кабинета показывает доставки по дням (стековые колонки по статусам,
чистый SVG без чарт-библиотек) с фильтром периода 7/14/30 дней, hover-тултипом
и табличным представлением; `GET /api/logs/stats?days=N` отдаёт тоталы и
подневную разбивку. Цвета статусов проверены на различимость при цветовой
слепоте (worst-pair CVD ΔE 12.4, порог 12).

### Аналитика (дашборд)

- **Success rate** как главная метрика + плитки отправлено/ошибки/в очереди.
- **Разрез по направлениям и по событиям** — сколько доставлено/ошиблось по
  каждому пикселю (Meta/TikTok) и по каждому типу события, с долей успеха.
- **Сводка ошибок** — топ причин отказов (истёкший токен, нет email/телефона,
  отклонение платформой…) с подсказкой как чинить, вместо чтения логов построчно.
- **Воронка событий** — объём успешных конверсий по типам событий.
- **Фильтры** над всем содержимым: по подключению, направлению, событию —
  масштабируют все блоки разом.
- Эндпоинты: `GET /api/logs/{stats,breakdown,errors,funnel,filters}` — все
  принимают `days`, `connectionId`, `destinationId`, `eventName`.

### Аналитика (отдельная страница «Аналитика») — детальная

`GET /api/logs/analytics` (те же фильтры) отдаёт всё для страницы:

- **Обзор со сравнением период-к-периоду** (дельта vs предыдущий равный период).
- **Тренд success rate** по дням (лайн-чарт).
- **Задержка доставки** от приёма события из CRM до отправки: среднее, медиана,
  p90 (по `createdAt → sentAt`).
- **Надёжность доставки** — с первой попытки / после ретраёв / провалено.
- **Качество матчинга** — покрытие email/телефон/external_id (ключевая метрика
  CAPI: чем выше, тем лучше платформа сопоставит конверсию) + предупреждение о
  конверсиях без единого идентификатора.
- **Выручка** — сумма переданных конверсий и средний чек по валютам.
- **Тепловая карта** активности: день недели × час (последовательная шкала).
- **Разрезы** по подключениям CRM и по маппингам.

Для этого в `DeliveryLog` хранятся `value`/`valueCurrency` и признаки
`hasEmail`/`hasPhone`/`hasExternalId` (заполняются при приёме события).

### Упрощённые интеграции

- **Пошаговый мастер** подключения CRM (тип → реквизиты → готово). Для OAuth
  amoCRM Redirect URI показывается прямо в форме, чтобы вставить его в настройки
  интеграции до ввода Client ID/Secret. Аналогичный **мастер для направлений**
  (Meta/TikTok): платформа → реквизиты с подсказками где взять Pixel ID и токен →
  проверка тестовым событием.
- **Авто-загрузка воронок и этапов** из CRM: в маппингах воронка и этап
  выбираются из выпадающих списков с названиями (не ручной ввод ID). amoCRM —
  `GET /api/v4/leads/pipelines`; Битрикс24 — `crm.category.list` +
  `crm.status.list`. Названия сохраняются в маппинге и показываются в таблице.
- **Авто-подписка вебхука amoCRM**: после OAuth платформа сама вызывает
  `POST /api/v4/webhooks` (события `add_lead`, `status_lead`) — не нужно вручную
  настраивать WEB HOOKS. Для токен-доступа — кнопка «Подписать вебхук».
- **Диагностика подключения** одной кнопкой: проверяет токен, доступ к CRM,
  подписку вебхука, факт приёма событий и наличие маппингов — и показывает, что
  именно сломано.

**Поток данных:**
CRM шлёт вебхук → событие сохраняется и ставится в очередь `crm-events` →
процессор находит активные маппинги по `pipeline_id`/`status_id` (amo) или
`CATEGORY_ID`/`STAGE_ID` (Б24), тянет из CRM контакт (email/телефон) и сумму
сделки → на каждый маппинг создаётся `DeliveryLog` с дедуп-ключом
(`mappingId:entityId:statusId` — одно событие не отправится дважды) и джоб в
очереди `delivery` → отправитель хеширует PII и шлёт событие в Meta/TikTok;
при ошибке — до 5 попыток с экспоненциальным backoff.

## Быстрый старт

Требуется: Node.js 20+, Docker.

```bash
npm install                # зависимости всех воркспейсов
docker compose up -d       # PostgreSQL + Redis
npm run db:push            # применить схему Prisma
npm run dev                # api (http://localhost:3001) + web (http://localhost:5173)
```

Откройте http://localhost:5173, зарегистрируйтесь — воркспейс создастся
автоматически. Дальше по шагам «Быстрого старта» на дашборде.

Конфигурация API — `apps/api/.env` (см. `apps/api/.env.example`).
Для приёма реальных вебхуков локально нужен публичный URL: поднимите туннель
(`ngrok http 3001` или cloudflared) и пропишите его в `PUBLIC_API_URL`.

## Подключение amoCRM

Вариант A — OAuth (рекомендуется):
1. В amoCRM: Настройки → Интеграции → Создать интеграцию (внешняя).
2. Redirect URI — возьмите из карточки подключения в кабинете CAPI
   (`{PUBLIC_API_URL}/api/connections/amocrm/callback`). Требуется публичный
   домен с HTTPS — `localhost` amoCRM не принимает (для разработки: туннель
   ngrok/cloudflared или вариант B).
3. Создайте подключение в CAPI, указав адрес аккаунта, Client ID и Client Secret,
   затем нажмите «Подключить OAuth».

Токены: access живёт сутки, refresh — 3 месяца и **ротируется при каждом
обновлении** (платформа сохраняет новую пару автоматически, с защитой от гонки
параллельных воркеров).

Вариант B — долгосрочный токен: настройки интеграции → вкладка «Ключи и
доступы» → «Сгенерировать токен» (срок до 5 лет, показывается один раз);
вставьте в поле токена при создании подключения.

Вебхуки: amoCRM → амоМаркет → кнопка «Веб-хуки» → добавьте URL из карточки
подключения, события — «Смена этапа сделки» (и «Добавление сделки» при
желании). amoCRM отключает вебхук после 100+ невалидных ответов за 2 часа
(невалидный = не 2xx или дольше 2 секунд) — платформа отвечает 200 мгновенно,
обработка идёт в очереди. `pipeline_id` из вебхука не гарантирован — воронка
сверяется по данным сделки из API. Лимит API: 7 запросов/сек на интеграцию.

## Подключение Битрикс24

1. Входящий вебхук (для чтения данных): Разработчикам → Другое → Входящий
   вебхук, права CRM. Полученный URL (`https://портал.bitrix24.ru/rest/1/токен`)
   укажите как адрес при создании подключения.
2. Исходящий вебхук (события): Разработчикам → Другое → Исходящий вебхук,
   события `ONCRMDEALADD`, `ONCRMDEALUPDATE` (и `ONCRMLEADADD`/`ONCRMLEADUPDATE`
   для лидов), URL обработчика — из карточки подключения в CAPI.
3. Необязательно: `application_token`, показанный при создании исходящего
   вебхука, можно указать в подключении CAPI — тогда каждое входящее событие
   дополнительно сверяется с ним (рекомендация Битрикс24).

## Направления

- **Meta**: Pixel ID + access token (Events Manager → Настройки →
  Conversions API → Сгенерировать токен; либо токен системного пользователя).
  `test_event_code` — для проверки в разделе «Тестовые события».
  `action_source` по умолчанию `system_generated` (значения `crm` не
  существует); для CRM-лидов платформа автоматически добавляет обязательные
  `custom_data.event_source="crm"` и `lead_event_source`. Версия Graph API —
  `META_GRAPH_API_VERSION` в .env (по умолчанию v25.0; Marketing API живёт
  ~год на версию — обновляйте). События принимаются не старше 7 дней.
  Server-to-server дедупликации у Meta нет — платформа сама не отправляет
  одно событие дважды (dedupKey).
- **TikTok**: Event Source ID + access token из TikTok Events Manager.
  Для CRM-событий создайте в Events Manager источник данных «CRM» и
  используйте его CRM Event Set ID; `event_source` по умолчанию `crm`
  (настраивается через `config.eventSource`: `web`/`offline`/`app`).
  Актуальные стандартные имена событий с мая 2025 — `Lead`, `Purchase`
  (прежние SubmitForm/CompletePayment работают как легаси).

Кнопка «Тестовое событие» в кабинете отправляет проверочный `Lead`.

## Маппинги

Маппинг = подключение CRM + этап воронки → событие + направление.
ID этапов: в amoCRM — `pipeline_id`/`status_id` (видны в URL раздела настроек
воронки), в Битрикс24 — `STAGE_ID` сделки (напр. `C1:WON`) и `CATEGORY_ID`.
Пустая воронка = любой воронке. Флаг «Передавать сумму» шлёт бюджет
сделки как `value` + валюту.

## Эффективность таргета (снижение CPA/CPM)

- **Качество матчинга**: кроме email/телефона/ID платформа ловит **click-id**
  (fbclid→fbc, gclid, ttclid, yclid) и доп. поля (имя/город/страна/индекс) из
  полей CRM (авто-детект по имени + явный `fieldMap` подключения), хеширует и
  шлёт в пиксели. Click-id — сильнейший сигнал сопоставления с кликом по рекламе.
- **Советник по эффективности** (`GET /api/logs/advisor`): health-score 0–100 и
  приоритизированные рекомендации, как поднять матчинг и снизить цену за
  конверсию (мало click-id, конверсии без идентификаторов, падения доставки,
  истёкшие токены, value-оптимизация). Панель на дашборде.
- **Алерты в Telegram** (`/api/alerts`): раз в 15 минут проверка проблем и push
  в Telegram (бот @BotFather + chat id). Дедуп по набору проблем, токен шифруется.
- **Направления**: Meta CAPI, TikTok Events API, **Google Ads** (Enhanced
  Conversions for Leads по gclid), **Яндекс.Метрика** (офлайн-конверсии по yclid).
  Выбор отправителя — через `SenderRegistry`.

## Безопасность

- **Токены шифруются в БД** (AES-256-GCM): access/refresh CRM, client secret,
  application_token Битрикс24, access token пикселей. Ключ — `ENCRYPTION_KEY`
  (в проде задайте `openssl rand -hex 32`). Старые plaintext-записи читаются
  прозрачно; при смене ключа ранее зашифрованные токены станут нечитаемыми.
- **Rate-limiting** (`@nestjs/throttler`): 120 запросов/мин на IP глобально,
  10/мин на вход и регистрацию (защита от перебора); вебхуки исключены.
- **helmet** для security-заголовков, `trust proxy` для корректного IP за
  обратным прокси, единый JSON-формат ошибок без утечки стек-трейсов.

## Деплой (Docker)

Полный стек — Postgres + Redis + API + фронт с **авто-HTTPS через Caddy**
(Let's Encrypt-сертификаты выпускаются автоматически). Фронт отдаёт SPA,
проксирует `/api` на бэкенд и терминирует TLS — CORS не нужен, отдельный
certbot не нужен. API при старте применяет схему к БД (`prisma db push`).

**На VPS с доменом** — одной командой (Ubuntu/Debian, домен должен указывать на
сервер, порты 80/443 открыты):

```bash
curl -fsSL https://raw.githubusercontent.com/FlyInTheSkyAndOfTheDarkness/CAPI/main/deploy-vps.sh | sudo bash
```

Скрипт ставит Docker, клонирует репозиторий, генерирует секреты в `.env.prod` и
поднимает стек. Домен задаётся в [deploy-vps.sh](deploy-vps.sh) и
[apps/web/Caddyfile](apps/web/Caddyfile).

**Вручную:**
```bash
cp .env.prod.example .env.prod          # секреты и домен
docker compose -f docker-compose.prod.yml --env-file .env.prod up -d --build
```

CI (GitHub Actions, `.github/workflows/ci.yml`) на каждый push собирает оба
приложения и гоняет тесты.

**Боевое подключение реального amoCRM** (Redirect URI, доступы, проверка живой
конверсии) описано пошагово в [GOING_LIVE.md](GOING_LIVE.md).
